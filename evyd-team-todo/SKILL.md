# Team TODO 管理

通过读写本地 Markdown 文件 `todo/team-todo.md` 管理团队待办事项。无外部 API 依赖。

## 语言规则

本 skill 的模板以中文为权威源。输出时跟随用户语言:

1. **检测用户输入语言** - 中文输入 → 中文输出;英文输入 → 英文输出;其他语言同理
2. **结构不变** - 前缀(📋✅❓📊)、分隔符(`|`)、编号、换行、分隔线、排序规则、标记(P0/P1/P2)严格不变
3. **只翻译自然语言部分** - 即固定词句(如尾句、标题行、统计行),任务内容、姓名原样保留
4. **字段名不翻译** - `Due:`、`Created:`、`No Due Date` 始终用英文,不随语言变

**英文输出示例(对照中文模板):**

| 中文模板 | 英文输出 |
|---|---|
| 📋 你的待办({N}条未完成): | 📋 Your TODOs ({N} pending): |
| 搞完了跟我说,我帮你删。 | Done? Tell me and I'll remove it. |
| ✅ 已给{姓名}加了待办:{内容} | ✅ Added TODO for {姓名}: {内容} |
| ✅ 已删除:{内容} | ✅ Removed: {内容} |
| ✅ 已完成:{内容} | ✅ Completed: {内容} |
| ✅ 已更新:{内容} | ✅ Updated: {内容} |
| ✅ 已更新:{内容} → [~] 进行中 | ✅ Updated: {内容} → [~] In Progress |
| ✅ 已追加子项到:{内容} | ✅ Added sub-item to: {内容} |
| ✅ 已为{姓名}开启每日推送 \| {HH:MM} \| {渠道} | ✅ Daily push enabled for {姓名} \| {HH:MM} \| {channel} |
| ✅ 已关闭{姓名}的每日推送 | ✅ Daily push disabled for {姓名} |
| 📋 推送状态: | 📋 Push status: |
| 启用 / 禁用 | enabled / disabled |
| 上次:成功 / 失败 | last: ok / error |
| ❓ 匹配到{N}条,你说的是哪个? | ❓ Matched {N} items, which one? |
| 回复编号即可。 | Reply with the number. |
| 📊 团队待办汇总 {date} | 📊 Team TODO Summary {date} |
| {N}条未完成({M}项逾期) | {N} pending ({M} overdue) |
| 全员统计:{X}条未完成 / {Y}项逾期 | Total: {X} pending / {Y} overdue |

## 数据文件

- **主数据:** `todo/team-todo.md` - 所有成员的 TODO(workspace 根目录下)
- **成员映射:** `todo/members.md` - 姓名 ↔ userid ↔ 角色

> 路径相对于 workspace 根目录 `/root/.openclaw/workspace/`。

## 数据格式

每条 TODO 一行(可选缩进子项):
```
- [ ] 任务内容 | Created:YYYY-MM-DD | Due:YYYY-MM-DD
- [ ] 任务内容 | Created:YYYY-MM-DD
- [~] 进行中任务 | Created:YYYY-MM-DD | Updated:YYYY-MM-DD
  - 子项备注
- [x] 已完成任务 | Created:YYYY-MM-DD | Done:YYYY-MM-DD
```

**核心字段(必填):**
- `Created`:YYYY-MM-DD 格式
- `Due`:可选,YYYY-MM-DD 格式

**可选扩展字段(用户自然使用时支持,不主动引导):**
- `[~]`:进行中(区别于 `[ ]` 未开始)
- `[x]` + `Done:YYYY-MM-DD`:已完成保留记录(默认仍是删除,仅用户/admin 明确要求保留时使用)
- `Updated:YYYY-MM-DD`:最后更新日期
- 缩进子项:TODO 下方 2 空格缩进的 `- ` 行,用于备注/拆解/待确认事项

**无其他字段**(无来源、无优先级、无备注字段)

详细格式规范见 [references/data-format.md](references/data-format.md)

## 权限模型

根据 `todo/members.md` 中的角色字段判断权限:

| 操作 | admin | member |
|---|---|---|
| 创建 TODO(给任何人) | ✅ | ❌ |
| 查看 TODO | ✅ 全员 | ✅ 仅自己 |
| 更新/完成 | ✅ 全员 | ✅ 仅自己 |
| 删除 TODO | ✅ | ❌ |
| 查看全员汇总 | ✅ | ❌ |

### 身份识别与自动注册

每次对话开始时,必须先完成身份识别,再进入任何操作流程。

**Step 1 - 精确匹配 userid:**
1. 从会话上下文中获取发送者的 userid(企微 userid、飞书 userid 等)
2. 在 `todo/members.md` 的 `userid` 列中做**精确匹配**
3. 命中 → 直接使用该身份,进入操作流程
4. 未命中 → 进入 Step 2

**Step 2 - 询问姓名:**
1. 回复用户:`你好,我还不认识你。你叫什么名字?`(英文:`Hi, I don't have you on file yet. What's your name?`)
2. 等待用户回复姓名后 → 进入 Step 3

**Step 3 - 姓名比对与注册:**
1. 拿用户提供的姓名去 `todo/members.md` 的 `姓名` 列匹配:
   - **完全相同** → 回复:`members 里已经有一位"{姓名}",是你吗?`
     - 用户确认是 → 把当前 userid 追加绑定到该成员记录,回复 `✅ 已绑定,以后我认识你了。`
     - 用户确认不是 → 要求用户提供一个可区分的显示名,然后注册为新用户
   - **相似但不完全相同**(如"张三"vs"张三丰")→ 列出相似成员让用户确认:`members 里有这些相似的名字:\n1. {姓名A}\n2. {姓名B}\n你是其中之一,还是新成员?`
     - 用户选择已有成员 → 绑定 userid
     - 用户确认是新成员 → 注册为新用户
   - **完全无匹配** → 直接注册为新用户
2. **注册新用户** = 写入 `todo/members.md`(角色 = member)+ 在 `todo/team-todo.md` 新建 `## 姓名` section + 回复 `✅ 注册成功,欢迎 {姓名}!`

**禁止:**
- 不得跳过 Step 1 直接用姓名猜测身份
- 不得在 userid 未命中时默认关联到已有成员
- 不得在未经用户确认的情况下绑定或合并身份

## 四个核心操作

### 1 布置任务

**触发词:** "给XX加个todo"、"给XX布置任务"、"新增待办"

**流程:**
1. 解析:目标成员、任务内容、截止日期(可选)
2. 校验权限:仅 admin 可创建
3. 读取 `todo/team-todo.md`
4. 在目标成员的 section 下追加:`- [ ] 任务内容 | Created:YYYY-MM-DD | Due:YYYY-MM-DD`(Due 可选)
5. 若目标成员 section 不存在,新建 `## 姓名` section
6. 写回文件
7. 回复确认

**回复格式(严格遵循):**

有 Due 时:
```
✅ 已给{姓名}加了待办:{任务内容} | Due:{YYYY-MM-DD}
```

无 Due 时:
```
✅ 已给{姓名}加了待办:{任务内容} | No Due Date
```

**禁止:**
- 不得省略 ✅ 前缀
- 不得添加额外的问句、建议或装饰文字
- 不得重复展示待办列表

### 2 查看待办

**触发词:** "我的待办"、"今天有什么任务"、"待办列表"

**流程:**
1. 识别当前用户身份
2. 读取 `todo/team-todo.md`
3. member:只展示自己的未完成 TODO
4. admin:展示自己的,并提示可用"汇总"查看全员

**输出格式(严格遵循,不得省略或改写任何部分):**

```
📋 你的待办({N}条未完成):

{编号}. {任务内容} | Due:{YYYY-MM-DD} {优先级标记}
{编号}. [~] {任务内容} | Due:{YYYY-MM-DD} {优先级标记}
{编号}. {任务内容} | No Due Date

搞完了跟我说,我帮你删。
```

**逐行规则:**
1. 第一行固定 `📋 你的待办({N}条未完成):`,N = `[ ]` + `[~]` 条目数(不含 `[x]`)
2. 空一行
3. 每条一行:`{编号}. {任务内容} | Due:{YYYY-MM-DD}` 或 `{编号}. {任务内容} | No Due Date`
4. `[~]` 进行中的条目在任务内容前加 `[~] ` 前缀;`[ ]` 未开始的不加前缀
5. 有 Due 的条目在行末追加优先级标记(见下表),无 Due 不追加
6. 缩进子项跟随父条目展示,不单独编号
7. 排序:Due 从近到远,无 Due 放最后
8. 空一行
9. 固定尾句 `搞完了跟我说,我帮你删。`
10. 如有 `[x]` 已完成条目,在尾句后另起一行:`─── 已完成({M}条)───` 然后逐行列出,无编号

**优先级标记(根据 Due vs 今天日期计算,不可手动编造):**

| 条件 | 标记 | 含义 |
|---|---|---|
| Due < 今天 | P0 | 逾期 |
| Due = 今天 | P1 | 今天截止 |
| Due = 明天 | P2 | 明天截止 |
| Due > 明天 | 无标记 | |
| 无 Due | 无标记 | |

**示例(假设今天是 2026-04-13):**
```
📋 你的待办(4条未完成):

1. 完成API文档 | Due:2026-04-12 P0
2. [~] 去NUHS | Due:2026-04-13 P1
3. 晚上去老巴刹 | Due:2026-04-14 P2
4. 更新 openclaw 版本 | No Due Date

搞完了跟我说,我帮你删。
─── 已完成(1条)───
给康宁团队分享运营后台 | Done:2026-04-14
```

**禁止:**
- 不得省略 📋 前缀
- 不得改写尾句
- 不得添加「高优/中优/低优」等数据中不存在的字段
- 不得显示 Created 日期(个人查看不需要)
- 不得添加额外的问句、建议或装饰文字
- 不得用 emoji(🔥🔴⚠️等)替代 P0/P1/P2 标记

### 3 自然语言更新

**触发词:** "XX做完了"、"PRD写好了"、"把截止日改到下周五"

**流程:**
1. 识别当前用户身份
2. 读取 `todo/team-todo.md`
3. 在该用户(member)或全员(admin)的 TODO 中模糊匹配最相关条目
4. 若唯一匹配:直接执行
5. 若多条匹配:列出让用户确认
6. 支持的操作:
   - **标记完成:** 默认直接从文件中删除该行;若用户/admin 明确要求保留记录,改为 `[x]` + `Done:YYYY-MM-DD`
   - **标记进行中:** 将 `[ ]` 改为 `[~]`,追加 `Updated:YYYY-MM-DD`
   - **改截止日:** 更新 Due 字段(YYYY-MM-DD 格式)
   - **追加子项:** 在条目下方添加缩进子项,追加 `Updated:YYYY-MM-DD`
7. 写回文件
8. 回复确认

**回复格式(严格遵循):**

标记完成(删除):
```
✅ 已删除:{任务内容}
```

标记完成(保留):
```
✅ 已完成:{任务内容} | Done:{YYYY-MM-DD}
```

标记进行中:
```
✅ 已更新:{任务内容} → [~] 进行中
```

改截止日:
```
✅ 已更新:{任务内容} | Due:{YYYY-MM-DD} → {YYYY-MM-DD}
```

追加子项:
```
✅ 已追加子项到:{任务内容}
```

多条匹配时列出确认:
```
❓ 匹配到{N}条,你说的是哪个?

1. {任务内容A} | Due:{YYYY-MM-DD}
2. {任务内容B} | No Due Date

回复编号即可。
```

**禁止:**
- 不得省略 ✅/❓ 前缀
- 不得添加额外的问句、建议或装饰文字

### 4 全员汇总

**触发词:** "汇总"、"全员进度"、"团队报告"、"今日报告"

**权限:** 仅 admin

**流程:**
1. 读取 `todo/team-todo.md`
2. 按成员分组统计:未完成数、逾期数
3. **严格按** [references/report-format.md](references/report-format.md) **的模板输出,不得省略或改写任何部分**

### 5 每日推送

**触发词:** "给XX开每日推送"、"关掉推送"、"推送状态"、"测试推送"

**权限:** 仅 admin

**机制:** 通过 OpenClaw cron job 定时生成个人待办摘要并投递。渠道无关--推送走 cron delivery 路由,不绑定具体渠道。

**流程(开启推送):**
1. 在 `todo/members.md` 中查找目标成员的 userid
2. 确认推送渠道(默认跟随 admin 当前通道,或由 admin 指定)
3. 确认推送时间(默认每天 08:30)
4. 确认语言(中文名 → 中文,英文名 → 英文,admin 可覆盖)
5. 创建 cron job(每人一个独立 job)
6. 立即触发一次测试
7. 回复确认

**Cron 配置要点:**
- `sessionTarget: isolated` - 隔离 session,不污染主会话
- `delivery.mode: announce` - 系统投递结果到目标渠道
- prompt 内严格锁定输出格式,禁止额外文字,确保 announce 原样投递

**推送消息格式:** 复用 2 查看待办的格式(含 ⚠️P0/P1/P2 标记、排序规则、已完成区)

**管理操作:**

| 操作 | 触发 | 说明 |
|---|---|---|
| 开启 | "给XX开每日推送" | 创建 cron job + 测试 |
| 关闭 | "关掉XX的推送" | disable 或删除 cron job |
| 改时间 | "把推送改到9点" | 更新 schedule |
| 测试 | "测试XX的推送" | 手动触发 cron run |
| 状态 | "推送状态" | 列出所有推送 cron job |

**回复格式:**

开启成功:
```
✅ 已为{姓名}开启每日推送 | {HH:MM} | {渠道}
```

关闭:
```
✅ 已关闭{姓名}的每日推送
```

状态查询:
```
📋 推送状态:

{姓名} | {HH:MM} | {渠道} | {启用/禁用} | 上次:{成功/失败}
{姓名} | {HH:MM} | {渠道} | {启用/禁用} | 上次:{成功/失败}
```

详细配置模板和 prompt 见 [references/daily-push.md](references/daily-push.md)

## 注意事项

- 每次写文件前先读取最新内容,避免覆盖并发修改
- 完成的 TODO 默认删除;仅用户/admin 明确要求保留时才用 `[x]` + `Done:`
- `[~]`、`[x]`、`Updated:`、缩进子项均为可选扩展,不主动引导用户使用
- 不依赖任何外部 API(企微待办、智能表格等均不使用)
- 优先级标记统一用 P0/P1/P2 纯文本,不使用 emoji 标记(🔥🔴⚠️等在部分渠道无法渲染)
